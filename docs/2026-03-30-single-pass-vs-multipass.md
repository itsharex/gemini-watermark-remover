# 单轮与多轮处理评估

## 背景

当前去水印流程同时承担了两类职责：

1. 定位 Gemini 可见水印的位置、尺寸和边距。
2. 对定位到的区域执行去水印处理。

其中多轮处理（multipass）的原始目标，是尝试覆盖重复叠加可见水印的极端场景。但近期对 `src/assets/samples/8.png` 的排查说明，多轮路径不仅增加了复杂度，也可能在首轮已经足够干净的情况下继续过处理，重新引入边缘残留。

## 当前职责拆分

### 1. 定位能力并不依赖多轮

当前代码中，真正负责定位与候选搜索的是：

- `src/core/geminiSizeCatalog.js`
- `src/core/watermarkConfig.js`
- `src/core/candidateSelector.js`

当前定位链路已经包含：

- 官方尺寸命中时的确定性锚点定位
- near-official 尺寸的 catalog 投影
- `size jitter`
- `local shift`
- `preview-anchor` 搜索
- adaptive fallback

多轮处理本身位于：

- `src/core/multiPassRemoval.js`
- `src/core/watermarkProcessor.js`

它的职责是对已定位区域重复执行 `removeWatermark`，而不是负责找位置。

### 2. 多轮当前带来的附加复杂度

多轮能力不仅是一个循环，还扩展出了整套状态与接口：

- `maxPasses`
- `passCount`
- `attemptedPassCount`
- `passStopReason`
- `passes`
- `removeRepeatedWatermarkLayers`
- `+multipass` source 标记

这些概念已经扩散到：

- 主处理流程
- SDK 导出与类型声明
- userscript runtime
- benchmark / fixed sample 导出脚本
- core / regression / sdk 测试

因此，多轮不是局部优化点，而是一个已经形成公开能力面的子系统。

## 当前样本上的观察

### 样本 `8.png`

`8.png` 的问题不是“没检测到水印”，而是：

- 标准模板与位置命中
- 首轮处理后梯度残留已经大幅下降
- 旧的 multipass 停止条件没有把这种情况视为“足够完成”
- 后续 pass 继续叠加处理，反而重新做坏边缘

这说明多轮当前会引入新的失败模式。

### 样本 `5.png`

`5.png` 是当前少数“看起来像是多轮可能有价值”的样本，但实际结果是：

- 单轮后仍有明显残留
- 多轮会尝试继续处理
- 但最终通常被 `safety-near-black` 或其他安全条件拦下
- 本地多轮并没有稳定、明确地把这类强残留样本清理干净

这意味着多轮的存在理由主要来自理论目标，而不是已经被当前本地算法稳定兑现的收益。

### 样本 `6.png` / `7.png`

这类样本在当前实现下：

- `maxPasses=1` 与 `maxPasses=4` 结果一致
- 首轮已经可以完成安全清理

这说明大部分正常样本并不需要多轮。

## 是否适合回到单轮

结论：如果产品目标是“稳定处理 Gemini 常规单层可见水印，复杂叠层样本交给外部 ML 服务”，则回到单轮是合理方向。

原因如下：

1. 单轮可以明显降低流程复杂度。
2. 单轮可以降低最坏路径计算量与页面端时延波动。
3. 单轮可以减少过处理和误伤风险。
4. 当前多轮并未稳定解决重复叠层可见水印。
5. 定位能力可以独立继续增强，而不需要依赖多轮。

## 鲁棒性判断

### 单轮更强的地方

- 行为更单调，可预测
- 更容易区分“定位失败”与“去除失败”
- 更容易在失败时明确转交外部 ML 服务
- 更适合作为 userscript / 页面端默认路径

### 单轮更弱的地方

- 对真实重复叠层可见水印，本地恢复上限会下降
- 一些极端 synthetic 或构造样本可能无法继续“搏多一轮”

但当前多轮并没有稳定兑现这部分收益，因此从整体体验看，单轮可能反而更鲁棒。

## 能否先把定位做扎实

这是回到单轮前最重要的前提。

建议把“定位能力”从“去除能力”里独立评估，目标分三层：

### 1. 官方原生尺寸

目标：

- 对 catalog 中全部官方尺寸，默认锚点直接命中

这部分理论上最适合做成确定性定位。

### 2. near-official / 缩放 / 预览图

目标：

- 用 catalog 投影 + size jitter + local shift + preview-anchor 稳定回收正确锚点

这部分属于轻量搜索问题，不属于多轮问题。

### 3. 异常或叠层样本

目标：

- 单轮定位和去除后如果残留仍明显，则明确转交外部 ML 服务

这部分不建议继续靠本地 multipass 膨胀复杂度。

## 推荐路线

### 方案 A：平滑收敛

1. 页面端与默认 SDK 路径改为单轮。
2. 保留定位搜索链路。
3. 将多轮降级为实验或内部开关。
4. 当单轮后残留超阈值时，转交外部 ML 去水印服务。

优点：

- 风险较低
- 可逐步验证定位能力
- 保留研究极端样本的入口

### 方案 B：彻底收敛

1. 删除多轮实现与公开接口。
2. 统一到“定位 + 单轮去除 + 外部 ML fallback”架构。

优点：

- 代码面最简
- 用户行为最稳定

代价：

- 迁移成本更高
- 需要先对定位和 fallback 做完整验证

## 建议的下一步

在决定是否删除多轮前，先补一组“定位专项”验证：

1. 官方尺寸全量定位回归
2. near-official 缩放定位回归
3. local shift 恢复范围验证
4. preview-anchor 定位专项验证
5. 定位失败与去除失败的分层诊断输出

如果这套定位验证稳定成立，再将默认主路径切到单轮，会更稳妥。
